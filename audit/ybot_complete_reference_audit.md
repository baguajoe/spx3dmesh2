# Y-Bot Complete Reference Audit — File-substitution investigation

Date: 2026-05-09
Scope: Determine whether `importGLB` (or any auto-effect on import)
substitutes the user's picked file with `/public/models/ybot.glb`.
Constraint: read-only, no patches.

---

## TL;DR

**No substitution happens.** `importGLB` loads the user's picked
file via blob URL; no error handler, fallback, or sibling system
swaps in `/public/models/ybot.glb`.

But the user's correction about the Model Picker label ("Y Bot —
Female Mixamo rig") is real and important: the file at
`public/models/ybot.glb` is **NOT a Mixamo Y-Bot**. It is **Mixamo's
Alpha avatar**, converted from FBX via Assimp. Its mesh names are
`Alpha_Joints` and `Alpha_Surface` — the exact names the user has
been seeing in DevTools snapshots of every imported Mixamo file.

The reason every imported Mixamo file looks "the same as Y-Bot" is
NOT because the editor substitutes ybot.glb. It is because:
1. The local `public/models/ybot.glb` is actually the **Alpha**
   character mesh (mislabeled as "Y Bot" in the Model Picker).
2. The user's Mixamo downloads are presumably ALSO using the Alpha
   avatar (default if user didn't switch characters at mixamo.com).
3. So the local "Y Bot" file and the user's imported files contain
   the same underlying Alpha geometry.
4. Without animation playback (Fix 3 not done), every Alpha mesh
   sits in bind pose. Different files, identical visible result.

## 1. `public/models/ybot.glb` — file metadata

```
Path:      /workspaces/spx3dmesh2/public/models/ybot.glb
Size:      3,007,692 bytes (≈ 3 MB)
Birth:     2026-04-28 16:42:25 UTC
Modify:    2026-04-28 16:42:25 UTC
Format:    glTF binary, version 2
```

### glTF JSON header (first 200 bytes)

```
glTF magic + version 2 + length 3007692
JSON{
  "asset":{"version":"2.0","generator":"Open Asset Import Library (assimp v5.3.0)"},
  "extensionsUsed":["KHR_materials_specular","KHR_materials_volume","FB_ngon_encoding"],
  ...
}
```

The generator string is **`Open Asset Import Library (assimp v5.3.0)`**.
This is the FBX-to-glTF conversion library — the file was originally
an FBX, then run through assimp to produce this GLB. **Mixamo's own
GLB exports use a different generator (`FBX2glTF` or `FbxExporter`).**
This is consistent with: someone downloaded Alpha as FBX from Mixamo,
ran it through assimp, committed the result.

### Names embedded in the file (extracted via `head -c 80000 | grep '"name"'`)

Mesh & material:
- `Alpha_Body_MAT`
- `Alpha_Joints`
- `Alpha_Joints_MAT`
- `Alpha_Surface`

Hierarchy roots:
- `RootNode`
- `mixamo.com`
- `mixamorig:Hips` + the full Mixamo skeleton (with Assimp's
  characteristic `_$AssimpFbx$_PreRotation` /
  `_$AssimpFbx$_Translation` synthetic nodes — fingerprint of an
  assimp FBX→glTF conversion).

### Geometry bounds (from glTF accessors)

Position accessor #0:
- `max: [97.34, 180.46, 20.37]`
- `min: [-97.34, -0.01, -16.05]`

So the mesh in local geometry space is ~194 × 180 × 36 units. That's
a **cm-units** Mixamo character. Once the Armature's 0.01 scale is
applied at runtime, you get a ~1.8m tall body — exactly the size
the user has been seeing in DevTools snapshots.

### Verdict

**The local `ybot.glb` is the Mixamo Alpha avatar in cm-units, FBX
origin, assimp-converted to GLB, mislabeled "Y Bot" in the Model
Picker UI.** It is not custom-generated by the editor; it is a stock
Mixamo character file with a confusing filename.

### Git history

```

```

That single commit added only the GLB binary — no companion src/
changes. Nothing in the editor was generated FROM this file at the
time it was added; it was added as a **test asset** for a separate

## 3. Does `importGLB` actually receive the user's blob URL?

Yes, traced line-by-line:

```
App.jsx:531    const url = URL.createObjectURL(file);
App.jsx:532    new GLTFLoader().load(
App.jsx:533      url,                              // ← user's blob URL
App.jsx:534      (gltf) => { ... },                // success
App.jsx:551      undefined,                        // progress
App.jsx:552      () => { setStatus(...); ... },    // error
App.jsx:556    );
```

`url` here is the **blob URL of the user-picked File**, created at
the moment of import. There is no second `URL.createObjectURL` in
the import path. The success callback receives the parsed result
of THAT URL.

```bash
grep -n "URL.createObjectURL" src/App.jsx
# 531  importGLB:        URL.createObjectURL(file)        ← user file
# 5136 Model Picker upload: URL.createObjectURL(file)     ← user upload, different code path
```

These are independent calls. Each creates a unique blob URL. They
cannot collide.

## 4. Does anything race or load `ybot.glb` in parallel with `importGLB`?

`loadModelToScene("/models/ybot.glb", ...)` is the only path that
fetches ybot.glb in the main editor. It fires:
- From `ensureWorkspaceMesh("rigging")` (user clicks Rigging tool).
- From the Model Picker Y-Bot button (user click).

If the user clicked Rigging earlier in the session and the GLTFLoader
call is still in flight when File → Import GLB runs, the Y-Bot
load's success callback fires after our cleanup pass and adds Y-Bot
to the scene. We covered this race in the previous diagnostic
(`audit/ybot_persistence_audit.md` hypothesis #2).

**But the race adds Y-Bot as a SECOND object alongside the imported
file, not in place of it.** The user's earlier DevTools snapshots
showed there was only ONE character in the scene at a time — so the
race is not active in their current test.

## 8. Why every imported file "looks like Y-Bot"

The mesh names `Alpha_Joints` and `Alpha_Surface` the user saw in
DevTools are **inside the imported Mixamo file itself**. They are
not injected by the editor. Mixamo bakes those names into every
download with the Alpha avatar.

The local `public/models/ybot.glb` ALSO contains `Alpha_Joints` and
`Alpha_Surface` (extracted above) because it, too, is the Mixamo
Alpha avatar (FBX→assimp→GLB, mislabeled "Y Bot" in the picker).

So:
- Local `ybot.glb` mesh = Alpha character.
- User's `Swinging.glb` (Alpha avatar) mesh = Alpha character.
- Visually identical when both sit in bind pose.

Animation playback is **not wired**: `App.jsx:166` imports
`createMixer` and `playClip` but neither is ever invoked. The render
loop at `App.jsx:1782` does not call `mixer.update`. SkinnedMesh stays
in bind pose (T-pose) regardless of which Mixamo clip the file
contains. Every Alpha-avatar Mixamo file therefore renders as
"Alpha in T-pose" — visually the same shape the user has been seeing
all along and calling "Y-Bot in T-pose".

## 9. The Model Picker mislabel

`App.jsx:5117`:
```js
{ url: "/models/ybot.glb", label: "Y Bot", thumb: "🦾", desc: "Female Mixamo rig" }
```

The file at this URL is the Mixamo **Alpha** avatar. Mixamo's actual
Y-Bot is a male robot-style character that does NOT have
`Alpha_Joints` / `Alpha_Surface` mesh names. Either:
- The file was renamed at commit time but the picker label was never
  corrected.
- Or the file was deliberately swapped from a Y-Bot download to an
  Alpha download, and the URL/label kept the original name.

Either way, the picker description is wrong: `/models/ybot.glb` is
the Alpha avatar (which IS female-shaped), but it is not the file
called "Y Bot" on Mixamo.

## 10. Recommended next diagnostic (decisive test)

Two cheap tests to discriminate the remaining possibilities:

### Test A — Network tab proof
Before File → Import GLB, open DevTools Network, filter `glb`. Click
File → Import GLB → pick a file that is *known not to be the Mixamo
Alpha avatar* (e.g. download Mixamo's "X-Bot" rig with the X-Bot
character explicitly selected, or any non-Mixamo GLB). Watch the
network panel.

- If `/models/ybot.glb` is fetched → some path is doing substitution.
  We hunt deeper.
- If only blob URLs (no `/models/ybot.glb`) → no substitution; what
  the user sees IS their picked file's geometry.

### Test B — Different-character proof
Download from Mixamo with a non-Alpha character (X-Bot or Y-Bot
explicitly), with any animation. Import. The visible mesh should
look like X-Bot/Y-Bot, not Alpha. If it still looks like Alpha →
substitution exists. If it looks like X-Bot/Y-Bot → no substitution,
Alpha visual was just because user's other downloads were Alpha.

## Summary

- `public/models/ybot.glb` = Mixamo **Alpha** avatar (FBX→assimp→GLB, mislabeled).
- `Alpha_Joints` / `Alpha_Surface` mesh names live INSIDE that file
  AND inside any user-downloaded Mixamo Alpha animation. Not injected
  by the editor.
- `importGLB` loads the user's file via a unique blob URL, has no
  fallback, no error path that swaps in ybot.glb.
- No mount-time `useEffect` / panel / hook fetches ybot.glb on its own.
- No race condition is currently active (DevTools showed one
  character).
- Animation playback is not wired up; every Mixamo file sits in bind
  pose; visually all Alpha downloads are indistinguishable.

The path forward is **Fix 3 (animation playback / AnimationMixer)**
and a small UX fix to either rename `public/models/ybot.glb` →
`alpha.glb` and fix the Model Picker label, OR swap the file for a
real Y-Bot download. Tests A and B above will confirm there's no
hidden substitution before that work begins.
