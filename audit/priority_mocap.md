# Mocap (Motion Capture) — Deep Audit

Read-only audit. No code modified.

## Verdict at a glance

- **Demo-readiness: YELLOW**
  - Single-person live-webcam mocap → driving Y Bot avatar in the bundled MocapWorkspace **does work end-to-end** (MediaPipe Pose CDN + face + hands hooks → AvatarRigPlayer3D rotation-based bone driver → Y Bot GLB at `/public/models/ybot.glb`).
  - However, several advanced toggles in the same UI silently fail or produce garbage: **MultiPersonMocap is a stub**, **DepthEstimator method calls don't match the API**, **BVH export writes near-zero data**.
  - As long as you stay on the default (single person, depth off, BVH ignored, JSON export only) it demos cleanly.

## Implementation approach

- **Library**: MediaPipe (BlazePose / FaceMesh / Hands) loaded from `cdn.jsdelivr.net` at runtime via `loadScript()`. No local mocap inference, no TensorFlow.js, no MoveNet.
- **Driving 3D**: Custom rotation-based driver in `src/front/js/component/AvatarRigPlayer3D.jsx` — computes parent→child limb direction from MediaPipe landmarks, calls `Quaternion.setFromUnitVectors(restDir, currentDir)` per bone (arms, forearms, thighs, shins, neck, spine), with OneEuro filtering per axis. Mixamo bone-name resolver tries multiple variants (`mixamorig:Hips`, `Hips`, `pelvis`, etc.).
- **Smoothing**: `src/front/js/utils/smoothPose.js` provides ONE_EURO / KALMAN / EMA / RAW pipelines. UI lets the user pick.
- **Foot plant**: `src/mesh/FootPlantSolver.js` — `solveFootPlanting()` and `getFootPlantStats()` are wired and called pre-export.
- **Depth estimation**: `src/mesh/DepthEstimator.js` — MiDaS v2.1-small ONNX via `onnxruntime-web` from CDN. Falls back to luminance pseudo-depth if model fails. Designed to "lift z" of pose landmarks. **NOT wired correctly to live capture (see bugs).**

## Input sources

- **Webcam (live)**: `getUserMedia({video:{width:1280,height:720,frameRate:30}})` — driven by MediaPipe `Camera` util.
- **Video file (recorded)**: `VideoMocapSystem.jsx` (713 lines, full-featured: drag+drop, fps-controlled extraction, scrub, JSON export). Mounted under the "Video" tab of the workspace.
- **Saved JSON sessions**: re-imported on the Playback tab; can also pull from `${VITE_BACKEND_URL}/api/motion-sessions`.

## What body parts are tracked

- **Body**: 33-point MediaPipe Pose, all standard landmarks.
- **Face**: 468-point FaceMesh via `useFaceMocap` — derives `jawOpen`, `leftEyeOpen`/`rightEyeOpen`, `leftBrowRaise`/`rightBrowRaise`. Drives `Jaw`, `LeftEye`, `RightEye`, `LeftBrow`, `RightBrow` bones in AvatarRigPlayer3D.
- **Hands**: 21-point MediaPipe Hands × 2 via `useHandMocap` — derives `pinch` and `fist` per hand, and exposes raw landmarks. Visualised on overlay; **not retargeted onto avatar finger bones** (no per-finger rig binding, only overlay drawing).
- **Holistic**: not used. `MP_HOLISTIC` URL is referenced once in `MultiPersonMocap.js` but never loaded.

## Output formats

- **JSON**: `mocap_<timestamp>.json` — raw landmark arrays per frame with timestamps, jawOpen/blink/brow metrics. Solid.
- **BVH**: **Broken/stub** — `downloadBVH(frames)` in `MocapRetarget.js` writes a fixed HIERARCHY then `frames.map(f => Array(6).fill(0).join(' '))`. Every frame is "0 0 0 0 0 0". The file opens in Blender but the figure stands still.
- **GLB**: only an `onExportGlb` prop hook in MocapWorkspace — no exporter actually wired through; pressing "GLB" shows the button but is a no-op unless the parent passes a handler. App.jsx does not pass one.
- **`.spxmotion`**: Generated by `VideoFaceMocap3DPanel.jsx` for face-only capture; supported by `convertClipToSPXMotion` / `convertRawBonesToSPXMotion` / `downloadSPXMotion` — **but those functions are imported in `ExportToPuppetButton.jsx` and DO NOT EXIST as exports in `SPX3DTo2DPipeline.js`**. The button would throw on click. (It's also not wired into App.jsx, so it can't actually be clicked from UI.)
- **Backend save**: `POST ${VITE_BACKEND_URL}/api/save-motion-session` — server-side persistence (no backend endpoint shipped in this repo).

## Calibration / setup

- No explicit T-pose calibration UI for live capture. `MocapRetargeter.bindTPose(skeleton)` exists in `MocapRetarget.js` but **is not called from the workspace** — `retargetEnabled` checkbox sets state but nothing reads it on the path that drives bones (AvatarRigPlayer3D ignores it; MocapRetargeter is not instantiated). It is dead UI for live capture.
- Smoother config (One Euro / Kalman / EMA / Raw + complexity Lite/Full/Heavy + minDetect/minTrack) is exposed and works.

## Real-time perf vs recorded playback

- Live FPS counter shown in HUD; depends on MediaPipe model complexity (Lite/Full/Heavy = 0/1/2). At 720p with Full, expect ~25–30 fps on a modern laptop.
- Recorded frames replay via `setInterval(1000/30/speed)` in PlaybackTab — works, scrubable.

## Error handling

- `getUserMedia` errors caught and shown via `setError(\`Camera error: ${err.message}\`)` in red banner.
- MediaPipe CDN load failure: `loadScript` rejects, error bubbles up to setError.
- DepthEstimator load failure: caught, falls back to luminance pseudo-depth without breaking — but the live wiring is broken anyway (see bug list).
- **No fallback when no camera permission**: button just shows error; no helpful instructions or simulated/uploaded-video fallback prompt.

## Recording / saving

- "Record" button starts/stops, pushes into `recordingRef.current[]` per frame with `{time, landmarks, jawOpen, leftBlink, rightBlink, browRaise}`.
- Exports work for JSON; broken for BVH; missing for GLB.

## Replay / scrubbing

- Playback tab loads JSON or backend session; provides play/pause, scrubber, 0.25×–2× speed select, BVH export (which is itself broken).

## Known bugs / placeholders / API mismatches

1. **`MultiPersonMocap._simulateDetection()` is a placeholder** (`src/mesh/MultiPersonMocap.js:213-230`). Comment literally says "For now, simulate with placeholder data" and "Only simulate 1 person without real ML". When the user toggles "Track multiple people" in the workspace, `MultiPersonMocap` is constructed with the WRONG arguments — `new MultiPersonMocap(videoRef.current, overlayRef.current, {...})` (workspace line 301) but the class signature is `constructor(options = {})`. So `videoRef.current` is being interpreted as `options`. Then `await mp.start()` is called, but the class only exposes `init()`, `startCamera()`, `stop()` — there is NO `start()` method. **Multi-person path will throw immediately.**
2. **`DepthEstimator` API mismatch**:
   - Workspace calls `estimator.init()` (line 192) but class only exposes `load()`. So `init is not a function` → silent break in dev console.
   - Workspace calls `depthRef.current.estimateDepth(videoRef.current)` (line 237) but class exposes `estimateFromVideo()` / `estimateFromCanvas()` / `estimateFromImage()`. Method does not exist.
   - Workspace calls `depthRef.current.liftPoseTo3D(lms, dm)` (line 240) — does not exist on the class.
   - All three calls are inside try/catch swallowing errors (`} catch(e) {}`), so user sees no error — but depth never works. The "✓ Ready" badge will never appear because `onReady` callback is hooked correctly to `load()` not `init()`, and `load()` is never called.
3. **`MocapRetarget.downloadBVH` writes zero motion data**: every frame is six zeros. BVH export looks like it works (file downloads) but plays back as a static figure.
4. **`fixFootSliding` uses `require('./FootPlantSolver.js')`** in an ES-module file (`src/mesh/MocapRetarget.js:215`). `require` is not defined in browser ESM — will throw if called.
5. **`ExportToPuppetButton.jsx`** imports `convertClipToSPXMotion`, `convertRawBonesToSPXMotion`, `downloadSPXMotion` from `SPX3DTo2DPipeline` — none of those functions are exported (`grep "^export" src/pipeline/SPX3DTo2DPipeline.js` shows only `BONE_MAP_3D_TO_2D`, `CINEMATIC_STYLES`, `SPX3DTo2DRenderer`, `SPX3DTo2DSkeletonRenderer`, `getStylesByCategory`, `getStyleCount`, default). Component is dead — but not actually wired into App, so user cannot trigger the crash.
6. **Retargeting toggle in workspace is dead UI**: `retargetEnabled` state is set but never used to wire a `MocapRetargeter` to the live data path; AvatarRigPlayer3D receives the prop but ignores it (the `retargetEnabled` prop is destructured but not referenced inside `AvatarRig`).
7. **Hands are tracked but not skinned**: hand landmarks are drawn on the overlay; no per-finger bone driving on the avatar (so the avatar's hands stay rigid in T-pose orientation while the user's fingers wiggle on screen).
8. **`videoRef` on a `<video>` element has duplicate `className` attributes** (`MocapWorkspace.jsx:400`: `className="mw-cam"` and `className="mw-mirror"`) — second overrides first, plus React will warn. Cosmetic only.
9. **Backend dependency**: "☁ Save" calls `${VITE_BACKEND_URL}/api/save-motion-session`. With no backend it fails silently or alerts an error. Same for the Playback session list.

## Integration end-to-end trace

**Path 1 — Live single-person body+face+hands** (this is the demo path):

1. User clicks Workspace tools → "MoCap" → `setMocapWorkspaceOpen(true)` → `<MocapWorkspace open={true} />` renders.
2. User clicks **▶ Start Capture** → `startCapture()` (line 295).
3. With multiPerson off: `loadScript('cdn.jsdelivr.net/npm/@mediapipe/pose/pose.js')` then `cdn.jsdelivr.net/npm/@mediapipe/camera_utils/camera_utils.js`.
4. `new window.Pose({...})` → `pose.setOptions({modelComplexity, smoothLandmarks:true, ...})` → `pose.onResults(handlePoseResults)`.
5. `new window.Camera(videoRef.current, {onFrame: () => pose.send({image: video}), width:640, height:480})` → `camera.start()`.
6. Browser prompts for camera. User accepts. Stream binds to `<video ref={videoRef}>`.
7. Per frame: pose.send → onResults → handlePoseResults runs OneEuroFilter on landmarks, depth lift attempt (silently fails), packs `frame = {landmarks, jawOpen, leftBlink, rightBlink, browRaise, handData}`, `setLiveFrame(frame)`.
8. `<AvatarRigPlayer3D liveFrame={liveFrame} avatarUrl="/models/ybot.glb" smoothingEnabled={false} />` — useFrame() loop reads liveFrame, looks up Mixamo bones via `findBone(skeleton, key)`, computes per-bone `Quaternion.setFromUnitVectors(restDir, currentDir)`, applies euler XYZ to bone rotations.
9. Y Bot animates in the right viewport. Face metrics drive `jaw.rotation.x`, `leftEye.scale.y`, `leftBrow.position.y`, etc.

This **chain works** and is the strongest demo asset.

**Path 2 — Multi-person**: BROKEN — throws on Start (constructor / `start()` mismatch). Do not enable.

**Path 3 — Depth-aware**: BROKEN — `init()` / `estimateDepth()` / `liftPoseTo3D()` don't exist on `DepthEstimator`; errors swallowed. Toggle does nothing functional.

**Path 4 — Video import**: `VideoMocapSystem.jsx` runs MediaPipe Pose over video frames; appears self-contained (713 lines) and reasonable, **but I did not verify the entire processing+playback loop end-to-end** in this audit.

**Path 5 — Face-only → 3D rig**: `VideoFaceMocap3DPanel.jsx` (218 lines) — driveFaceBones works on a passed-in `THREE.Skeleton` and exports `.spxmotion`. Self-contained. Needs to be passed a real skeleton to be useful — App.jsx doesn't currently render this panel.

## Demo readiness verdict — YELLOW

**Why not GREEN**: visible UI for Multi-Person, Depth, BVH export is exposed in the same workspace as the working demo path. If a Techstars judge clicks any of those toggles, multi-person throws, depth shows "⟳ Loading..." forever, and BVH downloads a zero-motion file. Retargeting checkbox does nothing.

**Why not RED**: the headline single-person live mocap → Mixamo Y Bot avatar driven by webcam pose+face+hands works convincingly. JSON record + JSON re-import + scrub-playback works.
